Alt Text: The Foundation

Alt text describes images for screen reader users and provides context when images fail to load. It's the single most important accessibility feature.

Good Alt Text Examples

✓ Concise and descriptive (usually under 125 characters)
✓ Describes purpose, not appearance ("Chart showing Q1 sales" not "Bar graph")
✓ Doesn't repeat surrounding text
✗ Empty alt text for decorative images: alt=""
✗ Starting with "Image of..." (screen readers already announce it's an image)

Alt Text Checklist

Image Type Alt Text Example
Informative Describe content alt="Sunrise over the ocean with pink clouds"
Decorative Empty string alt=""
Data/Chart Summary + link to data alt="Sales by region chart. See table below for data."
Link/Button Describe destination alt="Buy premium plan - save 50%"
Text in image Include the text alt="Poster: Web Accessibility Workshop, May 15"

Semantic HTML: Use Proper Tags

Semantic HTML is the foundation of accessibility. Use tags that describe meaning, and ARIA becomes unnecessary.

Semantic vs. ARIA

❌ Bad: Div with ARIA (avoid this) 
<div role="button" onclick="save()">Save</div>

Screen readers announce "button" but it's not actually a button. Keyboard users can't access it.

✅ Good: Semantic HTML (do this) 
<button onclick="save()">Save</button>

Browser handles everything: keyboard, screen reader, focus management.

Semantic Tags for Structure

  • <header>: Page or section header
  • <nav>: Navigation menu
  • <main>: Primary content (use once per page)
  • <article>: Self-contained content
  • <section>: Thematic grouping
  • <aside>: Sidebar or related content
  • <footer>: Page footer

Rule: Always use semantic HTML first. Only use ARIA to clarify or enhance, never to replace semantic HTML.

Keyboard Accessibility

Many users navigate with just a keyboard (mobility impairments, power users, screen reader users). Your site must work perfectly without a mouse.

Test Keyboard Navigation

Try using this page with only the Tab key (and Shift+Tab to go backward). Can you:

  • Navigate to all interactive elements?
  • See a visible focus indicator?
  • Activate buttons and links?
  • Tab out of all sections (no keyboard trap)?

Focus Management

Focus indicates which element is currently active (will receive keyboard input). Good focus management requires:

  • Visible focus indicator: Don't remove the outline
  • Logical tab order: Tab through elements in reading order
  • No keyboard traps: Users can always tab out
  • Focus management: Move focus when opening modals or dynamic content
Good: Visible focus indicator 
button:focus {
  outline: 2px solid #667eea;
  outline-offset: 2px;
}
✗ Bad: Removing outline without replacing it 
button:focus {
  outline: none; /* Never do this without replacement */
}

Color Contrast

Color contrast matters for users with low vision or color blindness. Text must have sufficient contrast with its background.

WCAG Contrast Ratios

Level Regular Text Large Text (18pt+)
AA (Standard) 4.5:1 3:1
AAA (Enhanced) 7:1 4.5:1

Beyond Color

Never use color alone to convey information:

✗ Red input border alone = error (what about colorblind users?)
✓ Red border + error icon + error message = clear error indication

ARIA (Accessibility Rich Internet Applications)

ARIA enhances HTML for complex interactions. But remember: First use semantic HTML, then use ARIA to clarify or enhance.

Common ARIA Attributes

Attribute Purpose Example
aria-label Label element when visible text unavailable <button aria-label="Close menu">×</button>
aria-describedby Link to description text <input aria-describedby="help"><p id="help">...</p>
aria-expanded Indicate if collapsible content is open <button aria-expanded="false">Show More</button>
aria-hidden Hide decorative elements from screen readers <div aria-hidden="true">→</div>
aria-live Announce dynamic content changes <div aria-live="polite">...</div>

Complete Accessibility Checklist

Images & Media

  • All images have meaningful alt text
  • Decorative images use alt=""
  • Video has captions and descriptions
  • Audio has transcripts

Structure & Navigation

  • Heading hierarchy is logical (no skipped levels)
  • Use semantic HTML: <header>, <nav>, <main>, etc.
  • Skip link to main content provided
  • Landmark regions properly defined

Forms

  • All inputs have associated <label> tags
  • Required fields marked as required
  • Error messages clear and associated with fields
  • Form is fully keyboard accessible

Color & Contrast

  • Text contrast at least 4.5:1 (AA standard)
  • Color not the only indicator (include text/icons)
  • Focus indicators clearly visible
  • No flashing or seizure-inducing content

Keyboard & Interaction

  • All interactive elements keyboard accessible
  • Logical tab order
  • No keyboard traps
  • Focus management in modals/dialogs

Testing

  • Tested with screen reader (NVDA, JAWS, VoiceOver)
  • Lighthouse accessibility audit: 90+
  • W3C HTML Validator: no errors
  • Browser testing: Chrome, Firefox, Safari, Edge

Tools for Accessibility Testing

  • axe DevTools: Automated accessibility checks
  • WAVE: Web Accessibility Evaluation Tool
  • Lighthouse: Built into Chrome DevTools
  • NVDA / JAWS: Screen readers for testing
  • Color Contrast Analyzer: Check contrast ratios
  • W3C HTML Validator: Semantic HTML validation
